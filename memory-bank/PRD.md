# SparkFlow 当前版本 PRD

> 最后更新：2026-03-09

本文档描述的是 SparkFlow 当前仓库对应的真实产品范围，而不是早期愿景版草案。凡与代码现状不一致的旧设想，均以下述内容和 [`architecture.md`](/Users/hujiahui/Desktop/VibeCoding/SparkFlow/memory-bank/architecture.md) 为准。

## 一、产品定位

SparkFlow 是一个面向知识内容创作者的移动端创作工具，目标是把“碎片灵感采集 -> AI 整理与生成 -> 提词拍摄”串成一个尽量短的闭环。

当前版本强调三件事：

- 尽快记录灵感，而不是先整理结构。
- 把散落碎片转成可编辑、可拍摄的口播稿。
- 在 App 内完成从脚本到提词拍摄的最小闭环。

## 二、目标用户

当前版本主要服务以下用户：

- 做知识口播、观点表达、个人 IP 内容的创作者。
- 有较多零散语音灵感，需要先收集再加工的人。
- 希望在手机端完成“记录 -> 生成 -> 拍摄”流程的轻量创作者。

## 三、当前版本目标

当前版本不是完整的创作者 SaaS，而是一个本地优先、单用户优先的 MVP。核心目标是验证以下闭环是否成立：

1. 用户能否方便地沉淀碎片灵感。
2. 系统能否把碎片快速整理成可用脚本。
3. 用户能否在同一 App 内进入提词拍摄。

## 五、核心用户流

### 1. 灵感采集流

用户可以通过三种方式沉淀灵感：

- 录制语音并上传。
- 手动输入文本碎片。
- 导入外部媒体链接音频。

语音类碎片进入后端后，会经过：

- 音频保存
- 异步转写
- AI 摘要
- 自动标签生成
- 向量入库

最终在碎片列表中可见，并可进入后续筛选、归类和生成流程。

### 2. 碎片整理流

用户可以围绕碎片做以下操作：

- 查看碎片列表和详情。
- 按文件夹归类。
- 查看标签与筛选结果。
- 批量选择碎片。
- 通过相似检索或灵感云图理解内容分布。

这一层的目标不是深度编辑，而是让用户尽快找到可用于生成脚本的素材组合。

### 3. AI 生成流

当前脚本生成统一走一条路径：

- 用户选择多个碎片并在生成页选择模式。
- 后端收集碎片、知识库命中和可选网页搜索结果。
- 后端调用本地部署的 Dify Workflow API。
- Dify 返回结构化结果后，后端回流创建脚本记录。

当前模式说明：

- `mode_a`：强调更强结构化和可直接拍摄的输出。
- `mode_b`：强调更自然、更贴近创作者表达的输出。

### 4. 脚本消费流

脚本生成后，用户可以：

- 查看脚本列表。
- 打开脚本详情。
- 更新脚本状态或内容。
- 删除不需要的脚本。
- 从脚本页进入拍摄页。

### 5. 提词拍摄流

当前版本已支持最小拍摄闭环：

- 进入拍摄页面。
- 使用相机录制。
- 在拍摄页使用提词能力。
- 拍摄完成后保存到系统相册。

当前目标是先保证“能拍、能存、能回看脚本内容”，而不是做复杂拍摄特效。

## 六、功能模块说明

### 模块 A：灵感碎片库

这是当前版本最核心的入口模块，承担素材沉淀与基础整理职责。

当前能力：

- 语音碎片上传与异步转写。
- 文本碎片创建。
- 外部媒体音频导入。
- 摘要与标签自动生成。
- 碎片列表、详情、删除。
- 文件夹归类与批量移动。
- Tag 聚合、过滤与相似检索。
- 灵感云图可视化。

当前价值：

- 降低灵感记录成本。
- 让用户从“原始语音”过渡到“可理解的碎片资产”。

### 模块 B：AI 编导与脚本生成

这是 SparkFlow 当前的第二核心模块，负责把碎片转成脚本。

当前能力：

- 多碎片聚合生成。
- `mode_a` 与 `mode_b` 均由统一 Dify 工作流生成。
- Dify 工作流 run 状态查询与脚本回流。
- 脚本列表、详情、修改、删除。

当前限制：

- 生成质量仍高度依赖碎片内容质量与 Dify workflow / prompt 设计。
- 移动端当前不展示工作流运行态细节，超时或失败仍以统一生成失败文案反馈。

### 模块 C：提词与拍摄

当前版本已经具备 App 内闭环拍摄能力，但仍以最小可用为目标。

当前能力：

- 从脚本页进入拍摄页。
- 提词展示。
- 相机录制。
- 视频保存到系统相册。

当前不追求：

- 复杂滤镜
- 美颜
- 跨应用悬浮提词
- 云端视频存储与剪辑

### 模块 D：知识库

知识库在当前版本更接近“后端已打底、前端待补强”的状态。

当前能力：

- 文档创建
- 文档上传
- 列表与详情
- 搜索
- 删除
- 独立向量空间存储

当前状态：

- 后端主能力已具备。
- 移动端入口仍偏占位，尚未形成成熟的用户工作流。

### 模块 E：每日灵感推盘

这是一个已具备后端能力、但前端消费尚未完全收口的模块。

当前能力：

- 定时任务已存在。
- 后端可聚合昨日碎片生成脚本。
- 已提供相关 API 与手动触发接口。

当前限制：

- 首页“今日灵感卡片”尚未稳定接入。
- Push 提醒还不是当前版本稳定对外能力。

## 七、当前版本的非目标

以下内容不应再被写成当前版本既有能力：

- 创作者广场 / Agent Marketplace
- Creator / 普通用户双身份生态
- 订阅他人方法论 Agent
- 跨应用悬浮提词器
- 完整竞品视频解析工具箱
- 云端视频留存与二次剪辑
- 完整成熟的知识库移动端管理体验
- 已稳定上线的每日推盘首页卡片与推送通知
- 已完成的 `mode_b` 个性化写作增强

## 八、当前版本成功标准

当前版本是否成立，主要看以下几个问题：

1. 用户能否顺畅完成至少一种灵感采集方式。
2. 采集后的碎片能否稳定进入“转写 / 摘要 / 标签 / 列表展示”链路。
3. 用户能否从碎片中选材并生成可阅读的脚本。
4. 用户能否从脚本页进入提词拍摄并保存视频。
5. 在本地联调环境下，前后端能否围绕这一闭环稳定协作。

## 九、后续优先级建议

结合当前代码状态，接下来更合理的产品优先级是：

1. 做完 `mode_b` 的历史风格增强闭环。
2. 把每日推盘卡片稳定接入首页主流程。
3. 决定知识库是否升级为真实移动端工作流。
4. 收口端到端验证和异常路径体验。
