# SparkFlow 前后端协作规范

> 最后更新：2026-03-09

本文档约定 SparkFlow 在“移动端 + FastAPI 后端”由不同成员并行开发时的协作方式。目标不是增加流程，而是降低等待、返工和联调失败。

## 1. 协作原则

- 以用户流为单位协作，不以“页面完成”或“接口完成”单独算完成。
- 以前后端契约为中心协作，不以口头约定或聊天截图为准。
- 先定义最小可联调闭环，再补细节和体验优化。
- 项目当前仍处于无老用户开发期，破坏性变更可以随前后端同步落地，但必须提前对齐契约和测试。
- 对异步后台链路，默认以后端任务态为准，不再假设创建接口返回时最终实体已经完成。

## 2. 角色边界

### 前端负责

- 页面结构、交互流程、加载态、空态、错误态。
- API 调用接入、客户端状态管理、本地暂存和设备能力调用。
- 基于契约先行完成 mock 页面与交互验证。

### 后端负责

- 业务规则、数据存储、任务调度、AI 编排、权限和数据一致性。
- API contract 实现、字段语义稳定性、错误码和可观测性。
- 为前端提供真实响应样例和异常场景说明。

### 双方共同负责

- 用户流拆解、接口契约、联调验收标准。
- 字段命名、可空性、枚举值、状态流转语义。
- 破坏性改动评估和变更同步。

## 3. 推荐工作流

每个功能按下面顺序推进：

1. 明确用户流
   - 例：录音上传片段 -> 查看转写状态 -> 生成脚本 -> 进入拍摄。
   - 至少写清楚入口、关键状态、成功路径、失败路径。
2. 先出接口契约
   - 后端先定义 request / response DTO。
   - 前端基于契约完成页面和状态流，不等待后端逻辑全部完成。
3. 前端先 mock
   - 优先打通交互、骨架屏、空态、失败态。
   - 未接真实接口前，不阻塞页面开发。
4. 后端实现真实能力
   - 按模块边界完成 `presentation -> application -> repository/service`。
   - 尽早给前端真实响应样例和错误样例。
5. 联调验收
   - 使用根目录 `bash scripts/dev-mobile.sh` 做本地联调。
   - 按真实用户流走通，不只检查接口是否返回 200。
   - 对任务态接口，必须额外验证轮询、失败和重跑。
6. 合并前复核
   - 契约、文档、示例、联调结果同步更新后再合并。

## 4. 契约定义规则

SparkFlow 当前以后端模块内 `schemas.py` 作为 API contract 单一事实源，协作时遵守以下规则：

- 新接口先补 `backend/modules/<module>/schemas.py`。
- `presentation.py` 必须显式声明 `response_model=ResponseModel[...]`。
- 前端以 `/docs` 或 `/redoc` 中的 OpenAPI 作为最终对照，而不是从代码里猜字段。
- 每个新接口至少补 1 个成功响应样例；涉及状态流转的接口，补 1 个失败或处理中样例。
- 如果接口返回的是任务句柄而不是最终实体，必须同时约定任务查询与重跑接口。
- 每个字段都要明确：
  - 是否必填
  - 是否可为 `null`
  - 枚举值范围
  - 时间字段格式
  - ID 是否稳定可缓存

### 响应约定

- 所有对外接口默认返回 `ResponseModel` 包裹结构。
- 删除接口统一返回 `200 + ResponseModel[None]`，成功时 `data=null`。
- 如果新增字段，优先选择当前领域语义清晰的名称。
- 如果必须做破坏性变更，需提前同步前端并一起修改，不为历史本地库、历史快照或早期调用入口保留过渡层。

### 任务态约定

- 媒体导入、脚本生成、每日推盘触发等后台链路，创建接口默认返回 `task_id` / `task_type` / `status_query_url`
- 任务真实状态以后端 `task_runs` / `task_step_runs` 与 Celery 执行为准；客户端只查询 `/api/tasks/*`
- 外链导入接口只返回任务句柄；媒体元数据和最终 `audio_file_url` 统一从 `GET /api/tasks/{task_id}` 的 `output` 读取
- 旧的 `GET /api/transcriptions/{fragment_id}`、`GET /api/fragments` 与 `GET /api/fragments/{fragment_id}` 读取接口已移除
- 前端通过 `GET /api/tasks/{task_id}` 观察整体状态
- 需要展示细分节点时，通过 `GET /api/tasks/{task_id}/steps`
- 失败补偿统一走 `POST /api/tasks/{task_id}/retry`

## 5. 任务拆分方式

每个需求建议同时拆出三类子任务：

### A. 用户流任务

- 目标页面或入口
- 成功路径
- 失败路径
- 验收口径

### B. 前端任务

- 页面与组件
- hook / API 接入
- mock 数据
- loading / empty / error 状态

### C. 后端任务

- route + schema
- use case / repository
- provider / scheduler / storage
- 测试与样例数据

不要把需求只拆成“前端开发”“后端开发”两个大块，否则联调时容易发现中间状态没人负责。

## 6. 联调约定

### 本地启动

- 优先使用仓库根目录命令：`bash scripts/dev-mobile.sh`
- 后端 API 端口固定为 `8000`
- Expo / Metro 端口固定为 `8081`
- 前端业务请求只能指向 `8000`，不能指向 `8081`

### 联调检查项

每个用户流至少检查：

- 正常成功路径
- 请求中 loading
- 空数据返回
- 字段缺失或 `null`
- 后端报错提示
- 异步任务中的处理中状态
- 任务态接口的 `queued / running / failed / succeeded`
- 重试或刷新后的恢复结果

### 日志协作

- 前端复现异常后，优先查看 App 内错误日志页。
- 后端可直接读取 `backend/runtime_logs/mobile-debug.log` 辅助定位。
- 不要只发截图，至少同步接口名、请求参数、设备环境、复现步骤。

## 7. 变更管理

以下变更必须在联调前主动同步：

- 字段改名
- 返回结构变更
- 状态枚举新增或调整
- 同步返回实体改为异步任务返回
- 鉴权方式调整
- 调度行为变化
- 环境变量或启动方式变化

同步内容至少包括：

- 影响范围
- 是否包含破坏性契约调整
- 前端是否需要同时改动
- 是否要更新 README / architecture / API 示例

## 8. 合并前清单

功能合并前，默认检查以下内容：

- 契约已经落到 `schemas.py` 或 OpenAPI。
- 前端没有写死临时 mock 结构。
- 成功态、失败态、空态都有人实际验证过。
- 启动方式或环境假设有变化时，README 已更新。
- 模块边界或核心流程有变化时，`memory-bank/architecture.md` 已更新。

## 10. 当前项目的执行建议

结合 SparkFlow 当前仓库形态，建议按下面方式落地：

- 前端以页面 / hook 为开发单元，优先保证用户流完整。
- 后端以 `modules/*` 为开发单元，禁止把新增业务继续写回早期的非模块化组织方式。
- API contract 以后端 `schemas.py + response_model + /docs` 为准。
- 联调统一走 `bash scripts/dev-mobile.sh`，避免各自手动启动导致环境不一致。
